# 05. 로컬에서 시놀로지 직접 연결 테스트, 점검, 트러블슈팅

이 문서는 로컬 개발 환경에서 시놀로지 NAS에 배포된 Spring Boot 및 JSP 애플리케이션에 직접 연결하여 테스트하고, 발생할 수 있는 문제들을 점검 및 해결하는 방법을 상세히 안내합니다.

## 1. 사전 준비 사항

*   `01.기본환경구성` 문서에 따라 SSH 키가 생성되고 시놀로지 서버에 공개 키가 등록되어 있어야 합니다.
*   `04.시놀로지 Docker Compose 환경설정` 문서에 따라 `docker-compose.yml` 파일이 시놀로지에 배포되고 컨테이너가 실행 중이어야 합니다.
*   로컬 환경에 `Mobaxterm` 또는 `터미널`이 준비되어 있어야 합니다.

## 2. 로컬에서 시놀로지 직접 연결 테스트

### 2.1. SSH 연결 테스트

가장 먼저 로컬 환경에서 시놀로지 NAS로 SSH 접속이 정상적으로 이루어지는지 확인합니다.

1.  **터미널 또는 Mobaxterm 실행**: 로컬 컴퓨터에서 터미널 또는 Mobaxterm을 실행합니다.

2.  **SSH 접속 시도**: 다음 명령어를 사용하여 시놀로지 NAS에 접속을 시도합니다.

    ```bash
    ssh -i ~/.ssh/id_rsa your_synology_user@your_synology_ip -p your_ssh_port
    ```

    *   `-i ~/.ssh/id_rsa`: `01.기본환경구성`에서 생성한 개인 키 파일의 경로를 지정합니다.
    *   `your_synology_user`: 시놀로지 NAS의 SSH 접속 사용자 이름입니다.
    *   `your_synology_ip`: 시놀로지 NAS의 IP 주소 또는 도메인입니다.
    *   `your_ssh_port`: 시놀로지 NAS의 SSH 포트 번호입니다 (기본값 22).

3.  **접속 확인**: 비밀번호 입력 없이 접속이 성공하면 SSH 연결은 정상입니다. 만약 비밀번호를 묻거나 접속에 실패하면 `3.1. SSH 연결 문제` 섹션을 참조하십시오.

### 2.2. 배포된 서비스 접속 테스트

시놀로지에 배포된 백엔드(Spring Boot) 서비스에 로컬 브라우저를 통해 직접 접속하여 정상 작동하는지 확인합니다.

1.  **웹 애플리케이션 테스트**: 웹 브라우저에서 다음 URL로 접속하여 Spring Boot 애플리케이션이 정상적으로 로드되는지 확인합니다.

    ```
    http://your_synology_ip:8080/?api_url=http://your_synology_ip:8080
    ```

    *   `your_synology_ip`: 시놀로지 NAS의 IP 주소 또는 도메인입니다.
    *   `8080`: Spring Boot 애플리케이션의 포트입니다.
    *   `api_url`: `index.jsp`에서 API 서버 주소를 파라미터로 받을 수 있도록 설계되었습니다.

    메인 페이지가 표시되고, 각 메뉴 클릭 시 해당 페이지로 이동하면 정상입니다.

## 3. 점검 및 트러블슈팅

### 3.1. SSH 연결 문제

*   **`Permission denied (publickey)` 오류**: 개인 키 권한 문제이거나, 시놀로지 서버에 공개 키가 제대로 등록되지 않았을 수 있습니다.
    *   로컬 개인 키 권한 확인: `chmod 400 ~/.ssh/id_rsa`
    *   시놀로지 서버의 `~/.ssh/authorized_keys` 파일에 `id_rsa.pub` 내용이 정확히 복사되었는지 확인합니다.
    *   시놀로지 서버의 `.ssh` 디렉토리 및 `authorized_keys` 파일 권한 확인: `chmod 700 ~/.ssh`, `chmod 600 ~/.ssh/authorized_keys`
*   **`Connection refused` 오류**: 시놀로지 NAS의 SSH 서비스가 비활성화되어 있거나, 방화벽에서 SSH 포트(기본 22)를 차단하고 있을 수 있습니다.
    *   시놀로지 DSM `제어판` > `터미널 및 SNMP`에서 SSH 서비스가 활성화되어 있는지 확인합니다.
    *   시놀로지 DSM `제어판` > `보안` > `방화벽`에서 SSH 포트가 허용되어 있는지 확인합니다.
*   **잘못된 사용자 이름 또는 IP 주소**: `your_synology_user` 또는 `your_synology_ip`가 올바른지 다시 확인합니다.

### 3.2. 컨테이너 실행 문제

서비스 접속이 안 되거나 오류가 발생할 경우, 시놀로지 Container Manager에서 컨테이너 상태를 확인합니다.

1.  **Container Manager 실행**: 시놀로지 DSM에서 `Container Manager`를 실행합니다.
2.  **프로젝트 탭 이동**: `프로젝트` 탭에서 `board-project`를 선택합니다.
3.  **로그 확인**: 각 서비스(db, redis, app)를 선택한 후 `로그` 탭을 클릭하여 오류 메시지를 확인합니다.

    *   **`db` 컨테이너 문제**: PostgreSQL 컨테이너가 시작되지 않으면 애플리케이션이 정상 작동할 수 없습니다.
        *   로그에서 `DB_PASSWORD` 관련 오류가 있는지 확인합니다. `.env` 파일의 비밀번호가 올바른지 확인하세요.
        *   `postgres_data` 볼륨 권한 문제일 수 있습니다. 시놀로지 파일 스테이션에서 해당 폴더의 권한을 확인합니다.
    *   **`redis` 컨테이너 문제**: Redis 컨테이너가 시작되지 않으면 애플리케이션에서 Redis를 사용하는 기능이 작동하지 않습니다.
        *   로그에서 오류 메시지를 확인합니다.
    *   **`app` 컨테이너 문제**: Spring Boot 애플리케이션이 시작되지 않는 경우.
        *   로그에서 데이터베이스 연결 오류(`SPRING_DATASOURCE_URL`, `USERNAME`, `PASSWORD`)가 있는지 확인합니다. `application.yml`의 환경 변수 설정과 `.env` 파일의 값이 일치하는지 확인하세요.
        *   `db` 또는 `redis` 컨테이너가 완전히 시작되기 전에 `app`이 시작될 수 있습니다. `docker-compose.yml`의 `depends_on` 설정이 올바른지 확인하고, 필요시 `restart: on-failure` 또는 `healthcheck`를 추가하여 의존성 문제를 해결할 수 있습니다.

### 3.3. 포트 충돌 문제

시놀로지 NAS에서 이미 8080 (애플리케이션) 포트를 사용 중인 경우 컨테이너가 시작되지 않을 수 있습니다.

*   **Container Manager 로그 확인**: 포트 충돌 관련 오류 메시지가 있는지 확인합니다.
*   **시놀로지 DSM에서 포트 사용 확인**: `제어판` > `네트워크` > `DSM 설정`에서 사용 중인 포트를 확인하거나, SSH로 접속하여 `netstat -tulnp | grep :8080` 명령어로 확인합니다.
*   **해결 방법**: `docker-compose.yml` 파일에서 포트 매핑을 변경합니다. 예를 들어, 애플리케이션을 8081 포트로 변경하려면 `ports: - "8081:8080"`과 같이 수정합니다.
