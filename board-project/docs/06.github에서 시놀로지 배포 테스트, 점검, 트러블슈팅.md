# 06. GitHub에서 시놀로지 배포 테스트, 점검, 트러블슈팅

이 문서는 GitHub Actions를 통해 시놀로지 NAS로 프로젝트가 성공적으로 배포되는지 테스트하고, 배포 과정에서 발생할 수 있는 문제들을 점검 및 해결하는 방법을 상세히 안내합니다.

## 1. GitHub Actions 배포 테스트

`03.GitHub Actions 환경설정` 문서에서 설정한 워크플로우가 정상적으로 작동하는지 확인합니다.

1.  **코드 변경 및 푸시**: 로컬 개발 환경에서 프로젝트 코드에 작은 변경사항을 적용합니다. 예를 들어, `src/main/webapp/WEB-INF/jsp/index.jsp` 파일의 타이틀을 수정합니다.

    ```jsp
    <%-- src/main/webapp/WEB-INF/jsp/index.jsp --%>
    <title>메인 페이지 - 배포 테스트</title>
    ```

2.  **변경사항 커밋 및 푸시**: 변경된 코드를 `main` 브랜치에 커밋하고 푸시합니다.

    ```bash
    git add .
    git commit -m "feat: Update JSP title for deployment test"
    git push origin main
    ```

3.  **GitHub Actions 워크플로우 확인**: GitHub 저장소로 이동하여 `Actions` 탭을 클릭합니다.
    *   `CI/CD Pipeline` 워크플로우가 자동으로 트리거되어 실행 중인 것을 확인할 수 있습니다.
    *   워크플로우를 클릭하여 `build-and-push` 및 `deploy` 작업의 진행 상황을 모니터링합니다.

4.  **워크플로우 성공 확인**: 모든 작업이 초록색 체크 표시와 함께 성공적으로 완료되었는지 확인합니다.

## 2. 배포 점검

GitHub Actions 워크플로우가 성공적으로 완료되었다면, 시놀로지 NAS에 배포된 애플리케이션이 최신 변경사항을 반영하여 정상적으로 작동하는지 확인합니다.

1.  **시놀로지 Container Manager 확인**: 시놀로지 DSM에 로그인하여 `Container Manager`를 실행합니다.
    *   `프로젝트` 탭에서 `board-project`의 상태가 `실행 중`인지 확인합니다.
    *   `컨테이너` 탭에서 `board-app` 컨테이너의 `최종 업데이트 시간`이 GitHub Actions가 완료된 시간과 일치하는지 확인합니다. 이는 새로운 이미지가 성공적으로 pull되고 컨테이너가 재시작되었음을 의미합니다.

2.  **웹 브라우저를 통한 접속**: 웹 브라우저를 열고 시놀로지 NAS의 IP 주소와 애플리케이션 포트(기본 8080)로 접속합니다.

    ```
    http://your_synology_ip:8080/?api_url=http://your_synology_ip:8080
    ```

    *   수정한 타이틀(`메인 페이지 - 배포 테스트`)이 웹 페이지에 반영되었는지 확인합니다.
    *   각 메뉴 클릭 시 해당 페이지로 이동하는지 확인합니다.

## 3. 트러블슈팅

### 3.1. GitHub Actions 워크플로우 실패

*   **로그 확인**: GitHub Actions의 `Actions` 탭에서 실패한 워크플로우를 클릭하여 상세 로그를 확인합니다. 어떤 단계에서 실패했는지, 오류 메시지는 무엇인지 파악합니다.
*   **Secrets 확인**: `03.GitHub Actions 환경설정`에서 설정한 GitHub Secrets (`DOCKER_HUB_USERNAME`, `DOCKER_HUB_ACCESS_TOKEN`, `SYNOLOGY_HOST`, `SYNOLOGY_USER`, `SSH_PRIVATE_KEY`, `SYNOLOGY_SSH_PORT`, `PROJECT_PATH`)의 값이 올바르게 설정되었는지 다시 확인합니다. 특히 `SSH_PRIVATE_KEY`는 줄바꿈 등을 포함하여 정확히 복사되었는지 중요합니다.
*   **Dockerfile 또는 `docker-compose.yml` 문법 오류**: 빌드 단계에서 실패했다면 `Dockerfile` 또는 `docker-compose.yml` 파일의 문법 오류를 확인합니다.
*   **네트워크 문제**: `deploy` 단계에서 SSH 연결 실패가 발생하면 시놀로지 NAS의 네트워크 설정, 방화벽, SSH 서비스 활성화 여부를 다시 확인합니다.

### 3.2. 시놀로지 배포 후 애플리케이션 문제

GitHub Actions는 성공했으나 시놀로지에서 애플리케이션이 정상 작동하지 않는 경우, `05.로컬에서 시놀로지 직접 연결 테스트, 점검, 트러블슈팅` 문서의 `3.2. 컨테이너 실행 문제` 및 `3.3. 포트 충돌 문제` 섹션을 참조하여 Container Manager에서 컨테이너 로그를 확인하고 문제를 해결합니다.

*   **최신 이미지 미적용**: `docker-compose pull` 명령어가 제대로 실행되지 않았거나, `docker-compose up -d`가 이전 이미지를 사용했을 수 있습니다. SSH로 시놀로지에 접속하여 `cd ${{ secrets.PROJECT_PATH }}` 후 `docker-compose down && docker-compose pull && docker-compose up -d` 명령어를 수동으로 실행하여 강제로 최신 이미지를 적용하고 재시작해봅니다.
*   **환경 변수 문제**: `.env` 파일의 환경 변수 (`DB_PASSWORD`, `DOCKER_HUB_USERNAME`, `PROJECT_PATH`)가 올바르게 설정되었는지 확인합니다. 특히 `docker-compose.yml`에서 참조하는 변수 이름과 `.env` 파일의 변수 이름이 일치하는지 확인합니다.

이 문서를 통해 GitHub Actions를 이용한 시놀로지 배포 테스트, 점검 및 트러블슈팅 방법을 숙지하여 안정적인 CI/CD 파이프라인을 운영할 수 있습니다.
